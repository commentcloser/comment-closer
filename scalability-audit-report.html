lib/prisma.ts:10-14 — bare new PrismaClient() with no connection pool settings. On Vercel serverless, each cold-started function opens its own connection. Default Prisma pool is 5 connections per function instance.

2. Webhook Processing (Meta/Facebook/Instagram)

2.1 Throughput Analysis

Each webhook invocation processes comments sequentially inside nested for loops:

for (entry of body.entry)        → sequential
  for (change of entry.changes)  → sequential
    await handleFeedComment()    → blocking

app/api/webhooks/facebook/route.ts:57-78

Per-comment processing time (estimated):

OperationTime
DB operations (~15 ops)~150ms
OpenAI sentiment analysis~500–2,000ms
OpenAI reply generation~1,000–5,000ms (up to 28s with web search)
Meta Graph API calls (3–5)~500–1,500ms
Total per comment~2–8 seconds (without web search)
Vercel function timeout: 10s default, no maxDuration export on webhook routes. If Meta sends a batch with 3+ comments, the function will timeout.

2.2 Burst Handling (100 webhooks in 1 second)

Meta batches multiple changes into a single webhook payload, but can also send separate requests. With 100 concurrent requests:

  1. Vercel spawns ~100 serverless function instances
  2. Each opens up to 5 DB connections = 500 connections (exceeds most DB provider limits of 20–100)
  3. 100 concurrent OpenAI API calls hit rate limits
  4. 100 concurrent Meta Graph API calls hit per-app rate limits
  5. Result: Mass connection failures, 429s from OpenAI, degraded Meta API responses

2.3 Race Conditions

Duplicate webhook for same comment — two instances both run upsert at facebook/route.ts:196. Both succeed (upsert is safe). But then both proceed to sentiment analysis and reply generation. The updateMany claim at line 380 catches this — only one will get count: 1. Correctly handled.

Missing HMAC signature verification — webhooks don't verify the X-Hub-Signature-256 header. Any party can send fake webhooks to trigger AI reply generation and Meta API calls.

3. AI Reply Generation (OpenAI)

3.1 OpenAI Rate Limits

Using gpt-5 (openai.ts:93, aiReplyEngine.ts:588):

Limit TypeTypical Tier 1Typical Tier 3Impact
RPM (requests/min)5005,000At 200 users with active pages, ~100 comments/min = 200 OpenAI calls/min (sentiment + reply)
TPM (tokens/min)30K300KEach sentiment ~100 tokens, each reply ~300 tokens
No retry logic — when rate-limited (429), the reply silently fails with 'Rate limit exceeded' at aiReplyEngine.ts:623. The comment is marked ai_failed with no automatic retry.

3.2 Meta Graph API Rate Limits

Meta imposes per-app and per-page limits:

  • 200 calls/user/hour for non-whitelisted apps
  • Posting replies counts against the limit
  • Webhook handler makes 2–4 Graph API reads + 1 write per comment

At 50 users with 5 pages each = ~250 pages. If each page gets 10 comments/hour = 2,500 comments × 3–5 API calls = 7,500–12,500 Graph API calls/hour. This will hit rate limits.

3.3 Queue Backlog

There is no queue. Everything is processed synchronously in the webhook handler. If OpenAI is slow or down:
  • Webhook handler blocks until timeout
  • Meta receives a timeout/error and retries the webhook (up to 3 times with exponential backoff)
  • This creates a cascade: retries arrive while the system is already overloaded

4.2 Overlap Risk

GitHub Actions cron runs every 5 minutes. If a tick takes >5 minutes (e.g., Meta API is slow), the next tick starts while the previous is still running. Both fetch the same comments (no locking).

The query only checks status: 'ai_generated', so comments won't be double-processed during normal operation. However, if the cron posts the reply but fails to update the DB before timeout, the next tick will try again (idempotency gap).

7. Ranked Bottlenecks (Will Break First → Last)

1. DB Connection Exhaustion During Webhook Bursts

Breaks at: ~30–50 concurrent requests

Fix: Use Prisma Accelerate or PgBouncer for connection pooling. Set connection_limit in Prisma config.

Effort: Easy (1–2 hours). Add ?pgbouncer=true&connection_limit=5 to DATABASE_URL, or switch to Prisma Accelerate.

2. Webhook Timeout (Synchronous Processing)

Breaks at: ~3 comments per batch, or any web-search reply

Fix: Return 200 immediately to Meta, push work to a background queue (Inngest, QStash, or Vercel Background Functions). Process asynchronously.

Effort: Hard (2–3 days). Requires architectural change to decouple ingestion from processing.

3. Missing Webhook HMAC Verification

Breaks at: First attacker

Fix: Verify X-Hub-Signature-256 header against FACEBOOK_APP_SECRET before processing.

Effort: Easy (30 minutes). Add HMAC check at top of POST handler.

4. Admin Dashboard OOM (Loading All Rows)

Breaks at: ~5K users or ~50K comments

Fix: Replace JavaScript .filter() timeline computation with SQL GROUP BY DATE_TRUNC('day', createdAt) aggregate queries at admin/users:145-179.

Effort: Easy (1–2 hours).

5. OpenAI Rate Limits (No Retry)

Breaks at: ~250 comments/minute

Fix: Add exponential backoff retry (3 attempts) on 429 responses. Implement a token bucket limiter.

Effort: Medium (half day).

6. Meta Graph API Rate Limits

Breaks at: ~100 active pages

Fix: Cache promotion_status results (the 2 Graph API calls at facebook/route.ts:154-194 for ad detection are often redundant). Remove unnecessary fetch calls when the webhook payload already provides promotion_status.

Effort: Medium (half day). Also batch Graph API calls where possible.

7. Cron Throughput (4 replies/min)

Breaks at: >20 scheduled replies per 5 minutes

Fix: Increase take to 100, parallelize with Promise.allSettled (batch of 5–10), add distributed lock.

Effort: Easy (1–2 hours).

8. ILIKE Full Table Scan

Breaks at: ~100K comments

Fix: Add PostgreSQL pg_trgm extension + GIN index on message and authorName. Or use full-text search (tsvector).

Effort: Medium (half day for migration + testing).

9. Dashboard Query Contention

Breaks at: ~30–50 concurrent users

Fix: Add Redis/Vercel KV caching for metrics (5–60s TTL). Metrics don't need to be real-time.

Effort: Medium (1 day).

10. Missing Indexes

Breaks at: Gradual degradation from 50K rows

Fix: Add the 6 missing indexes listed in Section 1.4 via Prisma migration.

Effort: Easy (30 minutes).
